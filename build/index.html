    Introduction

    The CoinMarketCap API is a series of RESTful JSON endpoints created to meet the demands of data scientists, application developers, and enterprise platforms. The endpoints include parameters added at the end of specific URLs.

    This API reference covers all technical documentation necessary to work with third-party platforms and applications. In addition, one of the documentation sections, API FAQs, also offers answers to common questions.

    This example API documentation page was created with Slate.

    Quick Start Guide

    For application developers ready to start using the CoinMarketCap API, here are a few steps to make your first API call.

    1. Sign up for a free account at pro.coinmarketcap.com. This is Coinmarketcap's live production environment with the latest market data. Choose the free Basic plan if it meets your requirements or upgrade to a paid plan.
    2. Copy your API Key. After you sign up, you'll see your Developer Portal account dashboard. Copy your API Key from the API Key box in the top left. With your key, make a test call. You may use the code examples below to make a test call with your chosen programming language. This example retrieves several active cryptocurrencies by market cap. It returns market values in US dollars.
    3. Replace the API Key with your own or use the test API Key b54bcf4d-1bca-4e8e-9a24-22ff2c3d462c for sandbox-api.coinmarketcap.com testing. If you use your own API key, use the API domain pro-api.coinmarketcap.com.
    4. After you've confirmed that your API key is working, read the rest of this API Reference and begin making your application!

    Authentication

    Acquiring Your API Key

    API Key validation is required for all HTTP requests. Register at the API Developer Portal to get an API Key if you don't have one yet.

    Using Your API Key

    Any server-side programming language can make HTTP requests to target the CoinMarketCap API. Requests should target the domain https://pro-api.coinmarketcap.com.You can provide your API Key in REST API calls with a custom header named X-CMC_PRO_API_KEY.

    Warning: API keys should not be visible to users and should not be stored in GitHub respositories! One way of solving this is setting environment variables on the server-side. Environment variables can be set from the command line.

    export API_KEY="your_api_key"

    Notice: You must replace your_api_key with your personal API key.

    Use your server-side language of choice to get your api key from the environment variable API_KEY.

    For example in Python:

    coinmarketcap_api_key=os.environ.get("API_KEY")

    Then, in the custom header set X-CMC_PRO_API_KEY equal to the local variable coinmarketcap_api_key.

    headers = {

    'Accepts': 'application/json',

    'X-CMC_PRO_API_KEY': coinmarketcap_api_key,

    }

    API Key Usage Credits

    Most API plans include a daily and monthly limit or "hard cap" to the number of data calls that can be made. Usage is tracked as API "call credits", which are incremented 1:1 against successful (HTTP Status 200) data calls made with your key with these exceptions:

    • Error responses, usage stats endpoints, and account management endpoints are not included in this limit.
    • Paginated endpoints: List-based endpoints track an additional call credit for every 100 data points returned (rounded up) beyond the 100 data point defaults.
    • Coinmarketcap's lightweight /map endpoints are not part of this limit and always count as 1 credit. For more details, see documentation for individual endpoints.
    • Bundled API calls: Numerous endpoints support resource and currency conversion bundling. Bundled resources are also tracked as 1 call credit for every 100 resources returned (rounded up). So, for example, 101 resources returned would round up to 2 calls. Optional currency conversion bundling with the convert parameter also increments an additional API call credit for each conversion requested beyond the first.

    Note: The start of your monthly subscription day and time defines the credit usage periods for your API subscription. For example, if your monthly subscription began on the 5th at 5:30 am, each month on the 5th at 5:30am you are billed for the month and your usage limits for the month are reset. However, usage limits for the the free Basic tier reset each day at UTC midnight and each calendar month at UTC midnight.

    Standards and Conventions

    Every HTTP request must contain the header Accept: application/json. Also, you should send an Accept-Encoding: deflate, gzip header to get data fast and efficiently.

    Best Practices

    This section contains several recommendations about efficiently utilizing the CoinMarketCap API for an enterprise application. These recommendations are especially beneficial if your application already has a large base of users.

    Use ID Instead of Symbol

    While utilizing common cryptocurrency symbols on the API is easy, it is brittle. Many cryptocurrencies have the same symbol. For example, three cryptocurrencies commonly refer to themselves with the symbol HOT. Additionally, cryptocurrency symbols frequently change with cryptocurrency rebrands.

    When fetching cryptocurrency data from a symbol matching several active cryptocurrencies, the API returns the one with the highest market cap. For this reason, whenever possible, use the permanent CoinMarketCap IDs. These IDs never change.

    Previously a call to the /cryptocurrency/map endpoint would give a full list of all active currencies mapped to the unique id property. However, this map endpoint does not return all the cryptocurrency symbols and ids.

    Notice: Any production code should utilize these IDs for cryptocurrencies, exchanges, and markets to future-proof your code.

    Use the Right Endpoints

    The endpoint /cryptocurrency/listings/latest requests paginated and ordered lists of all cryptocurrencies. However, the endpoint, /cryptocurrency/quotes/latest selectively requesting only the specific cryptocurrencies you require. Many endpoints follow this pattern.

    Implement Caching Strategy if Necessary

    Standard legal data safeguards are built into the Commercial User Terms that application developers should be aware of. These Terms help prevent unauthorized redistribution and scraping of CMC data. They are worded to allow legitimate local caching of market data for your application.

    If your application has a large user base and you are concerned with the API throttling and call credit limits of your subscription plan, consider implementing a data caching strategy. For example, instead of making a /cryptocurrency/quotes/latest call each time a user of your application needs to fetch market rates for specific cryptocurrencies, you could pre-fetch and cache the most recent market data for every cryptocurrency in your application's local database every 60 seconds. This would require just 1 API call, /cryptocurrency/listings/latest?limit=5000, every 60 seconds. Then, anytime a user of your application needs to load a custom list of cryptocurrencies you could pull this latest market data from your local cache without the burden of additional calls. This method of optimization is practical for API customers with large, demanding user bases.

    Code Defensively for Robust REST API Integration

    Code defensively whenever implementing any high availability REST API service for mission-critical operations. Since the API is versioned, any response format change or breaking request would only be introduced via new versions of each endpoint. However, existing endpoints may still introduce new properties over time.

    Recommended Best practices:

    • Parse the API response JSON as JSON and not through a regular expression or other means to avoid brittle parsing logic.
    • Parsing code should explicitly parse only the response properties required to guarantee new fields that may be returned in the future are ignored.
    • Add robust field validation to your response parsing logic. Wrapping complex field parsing, like dates, in try/catch statements can minimize the impact of unexpected parsing issues (such as the unlikely return of a null value).
    • Implement a "Retry with exponential backoff" coding pattern for your REST API call logic. So, if your HTTP request gets rate-limited (HTTP 429) or encounters an unexpected server-side condition (HTTP 5xx) your code would automatically recover and try again with a recovery scheme.
    • You can use one of the many libraries available; including, one for Node and another for Python.

    Reach Out and Upgrade Plan

    Reach out to api@coinmarketcap.com if you're uncertain about the best implementation of the CoinMarketCap API in your application or your needs outgrow the current self-serve subscription tiers. CoinmarketCap will review your needs and budget and may be able to tailor a custom enterprise plan that is right for you.

    CoinmarketCap ID Map

    The CoinmarketCap ID Map returns a mapping of all cryptocurrencies to unique CoinMarketCap ids. According to Best Practices, you should utilize CMC ID instead of cryptocurrency symbols. Use the CMC ID when calling the API with other endpoints and within your application. Each returned cryptocurrency maps to id returning other identifiers including name, symbol, and token_address. By default, cryptocurrencies with actively tracked markets on supported exchanges are returned. Passing listing_status=inactive returns a map of all inactive cryptocurrencies. Also, passing listing_status=untracked returns a map of registered cryptocurrency projects that are listed but do not yet meet methodology requirements to have tracked markets.     Listings Historical

    The Cryptocurrency Listings Historical endpoint returns a ranked and sorted list of all cryptocurrencies for a historical UTC date.

    Technical Notes

    • This endpoint has an identical format to the /cryptocurrency/listings/latest endpoint, but it retrieves historical daily ranking snapshots from the end of each UTC day.
    • Daily snapshots reflect market data at the end of each UTC day and can be requested as far back as 2013-04-28 (as supported by your plan's historical limits).
    • The "date" parameter can be passed as a Unix timestamp or an ISO 8601 date, but just the date of the timestamp will be referenced. Time should not be included with the ISO date format.
    • The Cryptocurrency Listings Historical endpoint retrieves paginated and sorted lists of all cryptocurrencies. If you require historical market data on specific cryptocurrencies, you should use /cryptocurrency/quotes/historical.

    By default, Cryptocurrencies are listed by cmc_rank. You may optionally sort based upon any of the following:

    cmc_rank: CoinMarketCap's market cap rank as outlined in the methodology.

    name: The cryptocurrency name.

    symbol: The cryptocurrency symbol.

    date_added: Date the cryptocurrency was added to the system.

    market_cap: market cap which equals the latest trade price times the circulating supply.

    price: latest average trade price across markets.

    circulating_supply: approximate number of coins in circulation.

    total_supply: the approximate total amount of coins in existence right now (minus any coins that have been verifiably burned).

    max_supply: the best estimate of the maximum amount of coins that will ever exist in the currency's lifetime.

    num_market_pairs: number of market pairs across all exchanges that trade each currency.

    volume_24h: 24-hour trading volume for each currency.

    percent_change_1h: each currency's 1-hour trading price percentage change.

    percent_change_24h: 24-hour trading price percentage change for each currency.

    percent_change_7d: each currency's 7-day trading price percentage change.

    This endpoint is available in the following API plans:

    • Standard (3 months)
    • Professional (12 months)
    • Enterprise (Up to 6 years)

    However, the following plans do not feature the Cryptocurrency Listings Historical endpoints:

    • Basic
    • Hobbyist
    • Startup

    Cache / Update frequency: On the next UTC day, 30 minutes after midnight, historical data for the last completed UTC day is available.

    Plan credit use: 1 call credit per 100 cryptocurrencies returned (rounded up) and 1 call credit per convert option beyond the first.

    CMC equivalent pages: historical daily crypto ranking snapshot pages such as this one on February 02, 2014.

    OHLCV Historical

    The OHLCV Historical endpoint returns historical OHLCV (Open, High, Low, Close, and Volume) data in addition to the market cap for any cryptocurrency using time interval parameters. Currently, hourly and daily OHLCV periods are supported. However, at this time, volume is only supported with daily periods.

    Technical Notes

    • Only the date of the timestamp is used for daily OHLCV. Therefore, it's recommended that you send an ISO date format such as "2018-09-19" without time for this "time_period".
    • For every "time_period" between your "time_start" (exclusive) and "time_end" (inclusive), one OHLCV quote will be returned.
    • If a "time_start" is not given, the "time_period" is calculated in reverse from "time_end" using the "count" parameter, which defaults to 10 results.
    • If "time_end" is not given, the endpoint defaults to the current time.
    • If you don't require every "time_period" between your dates, you can adjust the frequency that "time_period" is sampled with the "interval" parameter. For example, with "time_period" set to "daily" you can set "interval" to "2d" to get the daily OHLCV for every other day. You could also set "interval" to "monthly" to get the first daily OHLCV for each month or set it to "yearly" to get the daily OHLCV value for the same date every year.

    Implementation Tips

    • If you query for a specific OHLCV date, your "time_start" should specify a timestamp of 1 interval prior because "time_start" is an exclusive time parameter (as opposed to "time_end", which is inclusive to the search). As an exclusive time parameter, when you pass a "time_start", the results will be returned for the next complete "time_period". For example, if you are querying for a daily OHLCV datapoint for 2018-11-30, your "time_start" should be one day prior, "2018-11-29".
    • If you only specify a "count" parameter to return the latest OHLCV periods, your "count" should be 1 higher than the number of results you expect to receive. "Count" defines the number of "time_period" intervals queried, not the number of results to return. Also, "Count" includes the currently active time period, which is incomplete when working backward from the current time. So, the daily OHLCV with "count=1" refers to what has happened so far today. Therefore, to get the last daily OHLCV value available (yesterday in this case), pass "count=2". If you want two days ago, pass "count=3".
    • This endpoint supports requesting multiple cryptocurrencies in the same call. Please note, the API response will be wrapped in an additional object when requesting multiple cryptocurrencies from the same call.

    OHLCV Latest

Since the current UTC day is ongoing (still active), these values are updated often.     Version History

    The CoinMarketCap API employs Semantic Versioning in the format major.minor.patch. According to this versioning, the current major version is accessed from the API request path as /v1/. Unbreaking minor and patch updates to the API are released regularly. These may include new data points, endpoints, and API plan features which are always introduced without breaking previous minor releases and patches. Therefore, new properties will become available in our existing /v1/ endpoints. However, any breaking change will be introduced under a new major version of the API, and legacy versions will be supported indefinitely unless otherwise stated. Subscribe to our API Newsletter to get email updates monthly on CoinMarketCap API enhancements.

    To authorize, use this code:

    require 'kittn'

api = Kittn::APIClient.authorize!('meowmeowmeow')

    import kittn

api = kittn.authorize('meowmeowmeow')

    # With shell, you can just pass the correct header with each request
curl "api_endpoint_here" \
  -H "Authorization: meowmeowmeow"

    const kittn = require('kittn');

let api = kittn.authorize('meowmeowmeow');

    Make sure to replace meowmeowmeow with your API key.

    Kittn uses API keys to allow access to the API. You can register a new Kittn API key at our developer portal.

    Kittn expects for the API key to be included in all API requests to the server in a header that looks like the following:

    Authorization: meowmeowmeow

    You must replace meowmeowmeow with your personal API key.

    Get All Kittens

    require 'kittn'

api = Kittn::APIClient.authorize!('meowmeowmeow')
api.kittens.get

    import kittn

api = kittn.authorize('meowmeowmeow')
api.kittens.get()

    curl "http://example.com/api/kittens" \
  -H "Authorization: meowmeowmeow"

    const kittn = require('kittn');

let api = kittn.authorize('meowmeowmeow');
let kittens = api.kittens.get();

    HTTP Request

    GET http://example.com/api/kittens

    Query Parameters

    Parameter Default Description
    include_cats false If set to true, the result will also include cats.
    available true If set to false, the result will include kittens that have already been adopted.

    Errors

    This error section is stored in a separate file in includes/_errors.md. Slate allows you to optionally separate out your docs into many files...just save them to the includes folder and add them to the top of your index.md's frontmatter. Files are included in the order listed.

    The Kittn API uses the following error codes:

    Error Code Meaning
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The kitten requested is hidden for administrators only.
    404 Not Found -- The specified kitten could not be found.
    405 Method Not Allowed -- You tried to access a kitten with an invalid method.
    406 Not Acceptable -- You requested a format that isn't json.
    410 Gone -- The kitten requested has been removed from our servers.
    418 I'm a teapot.
    429 Too Many Requests -- You're requesting too many kittens! Slow down!
    500 Internal Server Error -- We had a problem with our server. Try again later.
    503 Service Unavailable -- We're temporarily offline for maintenance. Please try again later.
